# ESP32 双通道 CAN 报文分析仪 —— 分析层开发计划

> 前提：底层双通道（A/B）CAN 收发驱动已就绪并测试通过。本计划只覆盖 **分析层 / Web 控制 / 安全机制**，不再涉及硬件与驱动搭建。

---

## 1. 设计核心：数据模型

整个分析功能都围绕一张 **"按 (通道, ID) 唯一化"的表** 展开。把模型定对，后面所有功能都是在它上面读写。

每条记录（一个唯一 ID，分 A/B 两套表，或用 channel 字段区分）建议存：

| 字段 | 含义 | 用途 |
|---|---|---|
| `channel` | A 或 B | 全程区分双通道 |
| `id` / `is_extended` | 11/29 位 ID | 标准/扩展帧 |
| `dlc` | 数据长度 | 显示、解析 |
| `data[8]` | 当前 8 字节 | 显示 |
| `prev[8]` | 上一帧 8 字节 | 字节级对比 |
| `byte_change_ts[8]` | 每个字节最后一次变化的时间戳 | 字节高亮 + 淡出 |
| `bit_change_ts[64]`（可选/压缩存） | 每个 bit 最后变化时间 | 位级高亮 |
| `last_rx_ts` / `prev_rx_ts` | 最近两帧到达时间 | Delta Time |
| `rx_count` | 累计帧数 | 频率/活跃度 |
| `period_est` | 估算周期（滑动平均） | 静态抑制、频率分析 |
| `change_score` | 近 N 秒变化次数 | 活跃度热力、排序 |
| `flags` | static / blacklist / whitelist / pinned | 过滤逻辑 |
| `label` | 别名备注（"音量""车速"） | 人工标注 |

> 唯一 ID 表是固定大小哈希/数组（标准帧最多 2048 个，扩展帧需哈希表 + 上限）。这部分常驻 RAM，体量很小；**真正吃内存/带宽的是"全量录制"**，见第 8 节。

## 2. 系统架构

**双核分工**（关键，决定不丢帧、UI 不卡）：

- **Core 0 — 实时采集**：CAN A/B 中断/轮询收帧 → 打时间戳 → 投入无锁环形队列。绝不在这里做 Web/字符串处理。
- **Core 1 — 分析 + Web**：从队列取帧 → 更新唯一 ID 表 → 跑过滤/统计 → 通过 WebSocket 推给浏览器。

**Web 推送策略**（别把每一帧都推上去，WiFi 扛不住高总线负载）：

- 浏览器订阅后，后端以固定频率（约 **10–20 Hz**）推送"表快照增量"+"变化事件"，而不是逐帧。
- 字节/位高亮的"淡出"在**前端**用时间戳算（后端推 `byte_change_ts`，前端比对当前时间渲染颜色衰减），减少推送量。
- 通信用 **WebSocket + 二进制帧**（定长结构体）而非 JSON 文本，省带宽、CPU。控制指令（加黑名单、发送、改配置）走小 JSON 即可。

## 3. 功能清单

### A. 显示与高亮
- **唯一 ID 列表**：每个 (通道,ID) 只占一行，新数据原地刷新；A/B 分两个区或带通道标签/颜色。
- **字节级变化高亮**：某字节变化→标红/闪烁，停止变化后几秒内颜色淡出恢复（前端按 `byte_change_ts` 衰减渲染）。
- 🔧**位级变化高亮（补充）**：按键状态常打包在 bit 里（一个字节 8 个按键），整字节高亮会糊掉。提供"按位展开"视图，精确显示哪一 bit 翻转——对找按键信号极其有用。
- 🔧**多进制显示（补充）**：每字节可切 HEX / DEC / BIN / ASCII。
- 🔧**活跃度热力（补充）**：按 `change_score` 给 ID/字节上色，一眼看出"谁最活跃"，并支持按活跃度排序。
- 🔧**冻结/暂停（补充）**：一键冻结画面慢慢看，后台继续采集不丢帧。

### B. 过滤与抑制
- **静态抑制**：自动隐藏数据长期不变的 ID（如每秒 100 帧的静止状态帧），保持屏幕干净、防视觉疲劳。可设"判定为静止"的时长阈值。
- **黑/白名单**：黑名单隐藏、白名单只看这些。
- **按 ID/范围/通道过滤、按 ID 搜索**。

### C. 对比找包（核心逆向流程）
- **基线采集**：车机静止时录 5–10s 报文。
- **一键拉黑基线**：把基线里出现过的 ID 全部加入黑名单并隐藏。
- **触发定位**：之后按下被测按键（如音量+），屏幕只冒出新出现的一两条——就是目标控制信号。
- 🔧**快照 Diff（补充，比黑名单更通用）**：拍快照 A、拍快照 B，直接显示"新增 / 消失 / 数据变化"的 ID 集合。适合"操作前 vs 操作后"对比，不止于静止基线。
- 🔧**Pre-trigger 环形缓冲（补充，强烈推荐）**：常驻保留最近 N 秒（如 5s）历史。按下"标记"瞬间，能回看**触发之前**那几秒发生了什么——按键信号往往一闪而过，事后才回看根本抓不到。

### D. 时间与频率分析
- **Delta Time**：同一 ID 相邻两帧的时间间隔。平时不发、按下才发，或按下后发送变快，时间差一眼看穿。
- 🔧**周期估算 + 抖动（补充）**：滑动平均周期、最大/最小间隔，识别"周期帧 vs 事件帧"。
- 🔧**总线负载 / 帧率 / 错误计数（补充）**：每通道实时 fps、总线占用率%、TWAI 错误计数与 bus-off 状态。接车前先看总线健康，避免误判。

### E. 波形 / 信号
- **数据波形图**：选某 ID 的某几个字节生成折线图，像看心电图一样看车速/转速/音量的升降趋势。
- 🔧**多字节信号（补充）**：支持把相邻 2 字节合成 16 位值，可选大端/小端、有无符号，再画曲线（很多物理量是 16 位）。
- 🔧**滚动计数器 / 校验和自动识别（补充）**：自动标出"每帧 +1 的字节"（rolling counter）和疑似校验字节（CRC/异或/求和）。逆向时这俩最浪费时间，自动提示能省很多事。

### F. 录制与回放
- **录制**：把捕获的报文存成 `.csv` 或 `.asc`（含通道、时间戳、ID、DLC、数据），带通道标记。
- 🔧**触发录制（补充）**：满足条件（出现新 ID / 某 ID 变化）才开始落盘，省空间。
- **回放**：在线/离线重发录制内容，**按原始时间间隔**重放，用于欺骗车机、复现反应；可选重定向到 A 或 B。

### G. 发送（含安全）
- **🚨 发送总开关（TX Master Switch）**：全局总闸，**默认 OFF / 监听-only**。任何发送（单帧、周期、回放、扫描）都必须先过这道闸。UI 上做成醒目常驻状态条。
- **单帧发送**：手动改某 ID 的数据发出去，反向验证该 ID 是否真能控制车机；明确选 A 还是 B 发。
- 🔧**周期/循环发送（补充，必备）**：很多控制信号要持续重复发才生效。支持设定周期持续发某帧，可同时挂多条。
- 🔧**字节/位扫描（补充）**：把某字节从 0x00 扫到 0xFF（或某 bit 0/1 翻转），自动逐个发出，配合观察车机反应，暴力定位"哪个值触发哪个动作"。
- 🔧**发送序列/脚本（补充）**：按时间表依次发多帧（如"解锁序列"）。

## 4. 安全设计（单独强调）

这台设备能往车上发报文，安全要前置：

1. **默认监听-only（listen-only / silent）**：上电不发任何东西，包括不发 ACK，避免无意干扰总线。
2. **发送总开关默认 OFF**，且分通道独立使能（A 能发不代表 B 能发）。
3. **回放 / 扫描 / 周期发送前二次确认**，并显示"即将向 通道X 发送"。
4. **发送限速**，防手滑刷爆总线。
5. UI 顶部常驻**当前发送状态横幅**（红=可发送 / 绿=只监听），杜绝"以为没在发其实在发"。

---

## 5. 双通道 A/B 贯穿原则

A/B 必须在**显示、过滤、统计、导出、发送、回放**每一处都可区分、可独立操作：

- 显示：分区或通道色标 + 标签；可单独只看 A / 只看 B / 合并看。
- 统计：负载、fps、错误计数各自独立。
- 导出：每条记录带 channel 字段。
- 发送/回放：明确指定目标通道，且受该通道独立 TX 使能约束。

## 6. 分阶段里程碑（每阶段都可编译 + 台架测试）

| 阶段 | 内容 | 验收（不接车，台架可测） |
|---|---|---|
| **P1 Web 骨架** | WebSocket 实时帧流；唯一 ID 表；A/B 分区；TX 总开关(默认OFF/监听-only) | 浏览器看到 A/B 实时帧、唯一 ID 去重正确 |
| **P2 高亮+抑制+时间** | 字节&位变化高亮+淡出；静态抑制；Delta Time；总线负载/fps | 制造变化能高亮、静止帧能隐藏、间隔显示正确 |
| **P3 对比找包** | 基线采集→一键拉黑；快照 Diff；Pre-trigger 缓冲；ID 标注+持久化 | 模拟"操作前后"，diff 能定位到变化 ID |
| **P4 波形+信号** | 字节/16位曲线（端序、符号）；多进制；计数器/校验自动识别 | 喂递增数据能画出斜线、能标出计数器 |
| **P5 录制+发送** | `.csv/.asc` 录制；按时序回放；单帧发送；周期发送；字节扫描 | 录一段→回放，环回(A发B收)验证收发一致 |

> **台架测试技巧（无车）**：把 CAN-A 的 TX 接到 CAN-B 的总线上（或用自收/loopback 模式），让一路发、另一路收，就能在没有车机的情况下造出"双通道流量"，把上面每个阶段都验完。

## 7. 存储与吞吐注意

- **唯一 ID 表**常驻 RAM，体量小，无压力。
- **全量录制**才是瓶颈：高总线负载下 RAM 撑不了几秒。两条路线：
  - 边录边**流式下载到浏览器**（不落 flash，靠 PC 存）——简单，推荐先用这个。
  - 录到 **SD 卡 / SPIFFS**——离线但容量/写速受限。
- **回放精度**靠时间戳间隔；高精度重放需要在采集端就打好微秒级时间戳。
- **WiFi 是瓶颈**：高负载总线下，前端只接收"增量+采样"，原始全量靠录制文件，别指望浏览器实时收全部原始帧。
